#2: <html lang> structural fix — DEFERRED to dedicated session
   Auditor wanted non-JS crawlers to see correct `<html lang="zh-Hant">` etc.
   The R19 JS workaround (synchronous documentElement.lang setter in
   app/[lang]/layout.tsx) covers screen readers + JS-aware crawlers (Googlebot,
   Bingbot, etc. all render JS in 2026). The remaining gap is non-JS crawlers
   (some social media preview bots, niche search engines).

   The proper structural fix requires Next.js route-group migration:
     app/(public)/[lang]/layout.tsx  ← new root layout, owns <html lang={params.lang}>
     app/(public)/[lang]/page.tsx, about/, contact/, etc.
     app/(admin)/layout.tsx          ← new root layout, owns <html lang="en">
     app/(admin)/management/*
     DELETE app/layout.tsx           ← multi-root pattern requires no top-level layout

   Blast radius: every page route file path changes; metadata, OG, font, CSS
   imports rewire. Not safe to ship in autonomous-loop mode where I cannot
   easily detect regressions in the routing tree. Schedule for a dedicated
   session with full smoke test.
