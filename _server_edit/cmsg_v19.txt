#1 <html lang> baked as "en" for every language variant — WCAG 3.1.1 violation
   Root cause: app/layout.tsx reads `headers().get('x-lang')` set by middleware.
   With generateStaticParams + force-static (added in R18), pages prerender
   at BUILD time when middleware does not run. The result: `<html lang="en">`
   is baked into all 5 × N static variants, so /zh /cn /fr /es all ship lang="en".

   Pragmatic fix (not a full structural refactor): inline synchronous <script>
   in app/[lang]/layout.tsx that sets `document.documentElement.lang/dir`
   from the build-time-known lang param. Runs before paint, so:
     - Screen readers re-read lang on DOM observation (WCAG compliant)
     - Googlebot full-render picks up the corrected lang
     - JS-aware crawlers index pages with correct language signal

   Verified (curl per lang):
     /zh script sets documentElement.lang="zh-Hant" ✓
     /cn → "zh-Hans" ✓  /fr → "fr" ✓  /es → "es" ✓  /en → "en" ✓
   Trade-off: non-JS crawlers still see lang="en" in initial HTML. Full
   structural fix requires moving <html> from app/layout.tsx to [lang]/layout.tsx
   which breaks non-[lang] routes (e.g. /api). Deferred to dedicated session.
