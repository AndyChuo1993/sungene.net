Round-14 final audit caught a deploy-window 5xx burst (16 errors in one second

that window returned 500 because pm2 fork-mode reload is essentially stop+start.

#1 PM2 CLUSTER MODE — true rolling reload, zero deploy-window 5xx

     10000, kill_timeout: 5000, max_memory_restart: 900M

   - `pm2 reload sungene` now cycles workers one at a time; the surviving instance
     keeps serving while the other rebuilds
   - Memory budget: 7.3GiB total, 2 × ~400MB instances comfortable
   - Hammer-tested with 300 reqs over 10s while triggering rolling reload:
     **0 non-200 responses**

#2 deploy-atomic.sh — cluster-aware rolling reload + extended grace
   - `pm2 reload sungene --update-env` (cluster rolling, not stop+start)
   - `.next.previous-static` snapshot retention extended 1h → 24h so any
     lingering CDN/browser references to old chunk hashes can still resolve
     for a full day

#3 App Router error boundaries (auditor item)
