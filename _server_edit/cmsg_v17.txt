fix: round-17 — cluster-mode regressions + R16 audit gaps

R16 audit caught two cluster-mode regressions I introduced in R14p3 (PM2 fork→cluster
switch) plus a content depth claim I made on shaky measurements. All addressed.

#1 RATE LIMIT — was failing open under cluster mode
   PM2 cluster with 2 workers gave each worker its own in-process Map → effective
   limit was 2× intended, and round-robin meant a single IP attacker never
   accumulated enough hits in any one worker to trip. Auditor confirmed: 15
   consecutive POSTs from one X-Forwarded-For all returned 200.

   Fix: lib/rateLimit.ts now persists counters to a small JSON file under
   /tmp/sungene-ratelimit.json. Both PM2 workers read/write the same file. Race
   conditions exist (two workers can both increment in flight) but allow at most
   1-2 extra requests past the limit, acceptable for spam prevention. For strict
   consensus upgrade to Redis (Upstash free tier would do it).

   Verified (hammer test, 15 POSTs from same IP, limit=10):
     req 1-10: 400 (validation rejects but rate counter increments) ✓
     req 11-15: 429 ✓
   Cluster-shared limit now works.

#2 ISR CACHE — was MISS-MISS-MISS-MISS for /en and /en/about
   Same cluster mode bug: Next's in-process ISR memory cache lived per-worker.
   Worker A served /en and cached the result locally → Worker B receives the
   next /en request and has nothing cached → full SSR every time.

   Fix: next.config.mjs `experimental.isrMemoryCacheSize: 0` disables the
   memory tier; Next falls back to the on-disk cache, which is shared via fs
   between workers.

   Verified: /en attempt 1/2/3/4 = HIT/HIT/HIT/HIT ✓
